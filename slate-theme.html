Slate Theme Design System

Project Overview

The UX goal for the Slate Theme Design System was to create a component library devoid of extraneous functionality that our users did not need. We needed a system to ensure UI output was consistent for both web and non web based product lines. Developers needed the ability to access these reusable components and also be provided the necessary documentation for implementation.

Problem Statement

We needed to manage the UI needs of multiple teams, working on different screens, within one workflow? We were finding scenarios in which developers on the same team, given identical instructions were producing UIs that looked very different. What was causing this? Would we need to replace existing controls in lieu of new components that better serviced our users needs? What communication methods could we use to quickly educate developers with the new standards?

My Role:UI Designer

Tools I Used

  • Adobe XD (wireframes, prototypes, pixel comps w/ design specs)
  • Adobe Illustrator (pixel comps)
  • Visio(specifications)

  • Visual Studio (backlog maintenance)
  • Adobe InDesign (persona layouts)

The Solution

  • Phase 1: Establish the specifications
  • Phase 2: Create the library
  • Phase 3: Integrate

1. Establish the specifications. After a proof of concept was vetted we began by taking an inventory of all graphical assets and use cases. A specification document accounted for 1. Usage Patterns and 2. Visual Patterns. I was tasked to author Visual Patterns. The UX team became a full service team that would then create a custom library of reusable assets. Although it was an Angular based library the visual patterns were also extended to the non Angular based products. Third party components were replaced with custom controls that would allow us greater flexibility in styling and usage. My visual documentation was required to provide how the theme translated in multiple products.

2. Creating the library. The Slate component library was created with custom built controls. One of the major products lines was to be used as the model to absorb the new assets. There was a full refactor of the body layout and grid. We then established Communities of Practice workshops to educate developers on how to properly implement the new refined components.

3. Integration. As the library components began to be absorbed, I conducted daily Design Reviews with the our core service team developers and quality assurance engineers both on and off the UX team. A steady backlog was created and made visible to all teams who were migrating to the new theme.
